Git is Your Personal Desk

Git is the powerful software on your computer that acts as your project's history book. It tracks every change, letting you create drafts (**branches**), save milestones (**commits**), and rewind to any previous version without fear.

GitHub is The Cloud Library

GitHub is the website where you store your project online. It's a central hub for backing up your work and, more importantly, for collaborating with others. You can share your novel, and others can suggest edits (**pull requests**).

The Core Git Workflow (Your Desk)

This is the fundamental cycle you'll use constantly on your local machine.

1. Make Changes
Edit your files
2. Stage Files
`git add .`
3. Commit
`git commit -m "msg"`
4. Branch (Optional)
`git branch new-idea`

The GitHub Workflow (The Library)

This is how you collaborate with others using a remote repository.

1. Fork/Clone
Get a copy of the project
2. Create a Branch
Work on a new feature
3. Push to GitHub
`git push origin my-feature`
4. Open a Pull Request
Ask for your changes to be merged

Terminology Showdown

`git pull` vs `git fetch`

Both update your local repo with remote changes, but in different ways. `fetch` is safer as it lets you review changes before merging.

`git fetch`

Downloads changes but doesn't merge. Like getting mail delivered but leaving it in the mailbox.

`git pull`

Downloads changes AND merges them. Like getting the mail and immediately opening and sorting it on your desk. (It's `git fetch` + `git merge`).

`git merge` vs `git rebase`

Two ways to combine work from different branches. `merge` preserves history exactly as it happened, while `rebase` creates a cleaner, linear history.

`git merge`

Creates a new "merge commit" to tie the two branch histories together. The history is accurate but can look messy.

`git rebase`

Takes your changes and moves them to the end of the other branch. It rewrites history to be a straight line. Powerful, but use with care on shared branches.
