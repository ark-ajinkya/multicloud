Linux User Permissions

Beginner → Advanced r · w · x & sticky

A visual, single‑page reference to understand, read, and set permissions correctly for files and directories on Linux.

1) Who can access?

Every file has three classes of users:

  • u = owner (creator, or set via chown)
  • g = group (primary group of the file)
  • o = others (everyone else)
TIP: Change owner/group: sudo chown user:group file

2) What do r, w, x mean?

For files

  • read → view content
  • write → modify content
  • xecute → run as a program/script

For directories

  • read → list names (ls)
  • write → create/delete entries
  • xecute → traverse (enter with cd, access items)

Combined

  • Dir r+x required to ls with details
  • Dir w+x to add/remove files
  • File x is meaningless without r for scripts you interpret

4) Octal mapping

BitValueMeaning
r4read
w2write
x1execute

Example: chmod 640 file → u=6 (rw-), g=4 (r--), o=0 (---)

5) chmod — numeric & symbolic

Numeric

chmod 755 app.sh   # u=rwx, g=rx, o=rx
chmod 640 doc.txt # u=rw, g=r, o=---

Symbolic

chmod u+x,g-w,o=rx  script.sh
chmod a-rwx,u+rwX   -R project/

X adds execute only if target is a directory or already executable.

Recursive: chmod -R 750 dir/ Use with care

6) Ownership

chown user file
chown user:group file
chgrp group file

Only root can change owner. Any user can change group to one they belong to (depending on distro policy).

7) Default perms with umask

umask subtracts bits from the default creation mode (usually 666 for files, 777 for directories).

# show current mask
umask        # e.g., 0022

# effective defaults
files:      666 - 022 = 644  (rw-r--r--)
directories: 777 - 022 = 755  (rwxr-xr-x)

Collaborative env: umask 0002 → group‑writable by default.

8) Special bits (advanced)

BitOctalApplies toEffectSymbol
setuid 4--- Executable file Runs with file owner's UID rws in user exec position
setgid 2--- Executable / Directory Exec: runs with file GID; Dir: new files inherit directory group rwx with s in group exec
sticky 1--- Directory Only owner (or root) may delete/rename items inside t in others exec 
# examples
chmod 4755 /usr/local/bin/tool   # setuid
chmod 2775 /srv/shared           # setgid on dir (group inheritance)
chmod 1777 /tmp                  # sticky bit (public, safe deletes)

9) Sticky bit explained

The sticky bit (+t) on a directory prevents users from deleting or renaming files they don’t own even if the directory is world‑writable.

Where it’s used

  • /tmp, /var/tmp
  • Team drop‑folders
  • Shared upload directories

How to set

chmod +t shared/
chmod 1777 shared/

Both commands are equivalent; octal is explicit.

How it looks

$ ls -ld /tmp
 drwxrwxrwt ... /tmp
           ^ t in others exec
Remember: sticky bit on files is ignored on modern Linux. It’s meaningful only for directories.

10) Quick recipes

  • Make script executable: chmod u+x script.sh
  • Private key safety: chmod 600 id_rsa
  • Private dir: chmod 700 ~/secret
  • Shared project (group devs): chgrp -R devs proj && chmod -R 2775 proj

11) Security pitfalls

  • Over‑permissive 777 everywhere → avoid.
  • Executables with setuid root → audit necessity.
  • World‑writable dirs without sticky → users can delete each other’s files.
  • Scripts with setuid are ignored by most shells (not effective).

  2. Create /srv/teamshare, set owner root:team, and perms 2775.
  3. Inside, make files as alice and bob; verify group inheritance (ls -l shows team).
  4. Set sticky bit and world‑write on /srv/drop using 1777. Verify deletion behavior.

15) Mental model

Think of permissions as who (u/g/o) can do what (r/w/x) to which object (file vs directory). Special bits (setuid/setgid/sticky) tweak defaults for execution, inheritance, and safety in shared spaces.
